01 · PROBLEM

문제

우리 팀이 풀려는 문제를 한 문장으로 고정하여 기능이 과도하게 확장되는 것을 막습니다.

기존 원가계산서와 입찰공고문을 복사·수정하는 과정에서 발생하는 금액·수식·부가세·날짜·법령 누락을, 업로드 후 자동 검증과 공고문 초안 생성으로 줄인다.
수식·합계 오류

기존 엑셀의 행·열을 수정하면서 셀 참조가 깨지고 총액이 달라집니다.

부가세·제비율 오류

포함/별도 판단과 최신 제비율 확인을 사람이 반복합니다.

날짜 수정 누락

공고문 여러 위치의 날짜를 각각 고치다가 일부가 과거 값으로 남습니다.

검토 근거 부족

왜 경고가 발생했고 어떤 기준을 적용했는지 확인하기 어렵습니다.

02 · DEVELOPMENT GOAL

개발 목표(MVP)

3일 안에 완성 가능한 범위를 기준으로, 실시간 외부 연동보다 핵심 업무 흐름의 작동 여부를 먼저 검증합니다.

01
원가계산서 엑셀 업로드.xlsx 파일의 첫 번째 시트와 주요 헤더를 읽어 화면에 표시
구현
02
표준 항목 매핑·수정공사명, 항목명, 수량, 단가, 금액 열을 자동 인식하고 실패 시 직접 지정
구현
03
핵심 계산 검증수량×단가, 소계, 부가세, 총계를 다시 계산해 차이를 표시
구현
04
제비율·법령 체크샘플 기준정보 JSON을 이용해 일치·누락 여부와 적용 근거 표시
샘플 데이터
05
A→B 데이터 전달검증 완료 데이터를 브라우저 메모리 또는 JSON으로 공고문 화면에 전달
구현
06
공고 일정·문구 자동 반영공고일, 마감일, 개찰일, 금액을 한 번 입력해 미리보기에 반복 반영
구현
07
미리보기·내보내기HTML 미리보기, JSON 저장, 브라우저 인쇄/PDF 출력 제공
구현
1일차 · 데이터 입력 기반

프로젝트 골격, 엑셀 업로드, 헤더 매핑, 표준 데이터 스키마, 원가 항목 테이블.

2일차 · 검증과 화면 연결

재계산, 합계·부가세 경고, 샘플 제비율·법령 규칙, 검증결과 화면, A→B 전달.

3일차 · 공고문과 시연 완성

날짜 논리 검증, 공고문 미리보기, JSON·PDF 출력, 오류 수정, 샘플 시나리오 리허설.

⚠
3일 MVP에서 제외

실시간 조달청 API, 실제 AI API, OCR, HWP 자동생성, 사용자 권한·결재, 서버 DB, 사내망 연동은 후속 버전으로 넘깁니다. 화면에서는 ‘연동 예정’ 또는 샘플 추천 결과로만 표현합니다.

03 · SCREENS & NAVIGATION

화면

사용자는 6개 화면을 순서대로 이동합니다. 검증 오류가 있으면 다음 단계로 넘어가지 않고 해당 화면에서 바로 수정합니다.

SCREEN 01시작·업로드파일 선택 SCREEN 02매핑·데이터 확인열 지정 SCREEN 03검증결과오류 수정 SCREEN 04공고정보 입력날짜·조건 SCREEN 05법령 검토추천 확인 SCREEN 06미리보기·출력PDF·JSON
01
시작·원가계산서 업로드프로젝트 생성과 .xlsx 파일 선택
핵심 요소
공사/구매 유형, 파일 드래그 영역, 샘플 파일 불러오기
다음 조건
지원 형식과 파일 크기 확인 후 매핑 화면으로 이동
예외 처리
형식 오류, 빈 시트, 암호화 파일 안내
02
매핑·데이터 확인업로드 열과 표준 필드를 연결
핵심 요소
시트 선택, 헤더 행 선택, 항목명·수량·단가·금액 드롭다운
다음 조건
필수 필드가 모두 매핑되어야 검증 시작 가능
사용자 행동
자동 인식 결과를 확인하고 오인식 항목만 수정
03
원가 검증결과재계산·부가세·제비율·법령 경고
핵심 요소
항목 테이블, 차액, 통과/경고/오류 상태, 기준정보 버전
다음 조건
합계 불일치와 부가세 미선택 오류를 해소해야 공고문 단계 이동
처리 방식
오류 행을 직접 수정하면 전체 금액을 즉시 재계산
04
공고정보 입력기본정보·일정·입찰조건 입력
핵심 요소
공고명, 공고일, 마감일, 개찰일, 계약기간, 계약방법
검증 규칙
공고일 < 마감일 ≤ 개찰일, 필수 날짜 누락 금지
자동 처리
원가검증에서 확정된 공사명과 총액 자동 채움
05
법령·필수조항 검토규칙 기반 추천 결과 확인
핵심 요소
필수 법령, 추천 근거, 적용/제외 체크, 담당자 메모
MVP 방식
laws.json의 공사유형·금액구간 규칙으로 샘플 추천
후속 확장
조달청 유사사례와 AI 추천 API를 동일 영역에 연결
06
공고문 미리보기·내보내기최종값 확인과 결과 저장
핵심 요소
문서형 미리보기, 자동채움 강조, 검증 체크리스트
출력
브라우저 인쇄/PDF, 데이터 JSON, 검증 결과 CSV 또는 엑셀
차단 조건
총액 오류, 날짜 오류, 필수 법령 미검토 시 출력 버튼 비활성화
04 · ROLES

역할

4명 기준 역할안입니다. 인원이 적으면 기획·QA와 개발 역할을 겸임하되, 데이터 스키마와 최종 검증 책임자는 분리합니다.

PO

기획·업무 담당

문제 정의, 화면 우선순위, 샘플 원가계산서·공고문 제공, 기준정보와 합격 기준 확정.

A

도구 A 개발

엑셀 파싱, 매핑, 원가 재계산, 부가세·합계·제비율 검증, 검증 데이터 생성.

B

도구 B 개발

공고정보 폼, 날짜 검증, 법령 체크, 문서 미리보기, PDF·JSON 내보내기.

QA

통합·QA·발표

A→B 연계, 샘플 테스트, 오류 목록 관리, 배포, 시연 시나리오와 발표자료 정리.

05 · COLLABORATION

협업 방식

도구 A와 도구 B를 병렬로 만들되, 첫날 오전에 공통 데이터 구조를 먼저 확정해야 마지막 날 통합 실패를 줄일 수 있습니다.

TRACK A

원가 업로드·검증

엑셀 파일을 표준 JSON으로 변환하고 금액 오류를 판정합니다.

  • 입력: 원가계산서 .xlsx
  • 출력: validatedProject 객체
  • 완료 기준: 샘플 정상·오류 파일 처리
TRACK B

공고문 생성·출력

고정된 샘플 JSON으로 먼저 개발한 뒤 도구 A 결과를 연결합니다.

  • 입력: validatedProject 객체
  • 출력: 공고문 HTML·JSON·PDF
  • 완료 기준: 날짜 검증과 미리보기
INTEGRATION

공통·QA

데이터 스키마, 공통 UI, 샘플, 테스트 목록과 배포를 관리합니다.

  • 매일 2회 10분 통합 점검
  • 오류는 GitHub Issue로 즉시 기록
  • 3일차 오전부터 기능 동결
WORK RULE

작업 규칙

  • main: 시연 가능한 상태만 병합
  • feature/excel, feature/notice: 기능별 브랜치
  • 작은 단위로 커밋하고 Pull Request에 화면 캡처 첨부
  • 공통 객체 필드 변경은 팀 전체 합의 후 반영

08 · LEARNING CONCEPTS

필요한 학습 개념

모든 개념을 깊게 공부한 뒤 시작하기보다, 구현에 필요한 순서로 바로 적용하면서 익히는 방식이 적합합니다.

01
브라우저 DOM과 이벤트

버튼 클릭, 파일 선택, 폼 변경, 화면 전환과 오류 메시지 표시.

02
JavaScript 객체·배열

엑셀 행을 표준 객체로 변환하고 map, filter, reduce로 계산.

03
File API와 SheetJS

사용자가 선택한 엑셀을 ArrayBuffer로 읽고 시트 데이터를 JSON으로 변환.

04
데이터 스키마와 매핑

서로 다른 열 이름을 itemName, quantity, unitPrice 등 공통 필드로 통일.

05
규칙 기반 검증

합계 차액, 부가세 선택, 날짜 선후관계, 필수 법령 누락을 조건문으로 판정.

06
상태 관리와 로컬 저장

현재 단계, 검증 상태, 사용자 수정값을 하나의 상태 객체와 localStorage로 관리.

07
문서 템플릿 생성

데이터를 HTML 문서의 반복 위치에 안전하게 삽입하고 인쇄용 CSS를 구성.

08
Git 브랜치와 병합

기능별 브랜치, 작은 커밋, Pull Request, 충돌 해결과 배포 흐름.

09
API·비동기 처리 기초

후속 버전의 fetch, async/await, 오류 처리, CORS와 서버리스 프록시 개념.

10
보안·감사 추적

민감정보 최소화, API 키 비노출, 사용자 수정·판단 이력 기록의 필요성.

✓
학습 우선순위

1일차에는 DOM·객체·SheetJS·스키마, 2일차에는 검증 규칙·상태관리, 3일차에는 문서 출력·Git 통합·API 확장 개념 순으로 학습합니다.
