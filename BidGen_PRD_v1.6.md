## 1. 배경 (Background)

우리 회사는 조달청을 통해 입찰공고를 진행하고 있으며, 공고문 작성 → 원가계산 → 견적서 입력에 이르는 계약업무 전 과정이 **담당자 개인의 수작업과 판단**에 크게 의존하고 있다. 이 과정에서 반복적으로 오류가 발생하여 재공고, 예산낭비, 재작업 등의 비효율이 누적되고 있다.

본 프로젝트는 이 반복 오류의 **표면 증상이 아닌 근본원인**을 인터뷰 기반으로 진단하고, 이를 해소하는 **2단계 파이프라인 도구**를 Phase 1 파일럿(스탠드얼론 HTML)으로 설계한다.

```
[도구 A] 원가설계서 표준검증 도구
   · 견적서 금액 검증
   · 제비율 적용 검증
   · 적용법령 검토
   · 수식(계산 로직) 검토
        ↓  (검증 완료된 원가설계서 데이터 전달)
[도구 B] 입찰공고문 자동작성 도구
   · 원가설계서 기반 공고문 자동 생성
   · 날짜/기한 단일 입력 → 다중 위치 자동 반영
   · AI(Claude API) 기반 적용법령 누락·보완 추천 (공사 내용 학습형)
```

두 도구는 순차적으로 연결되는 하나의 워크플로우이며, 도구 A의 검증 완료 결과가 도구 B의 입력값이 된다.

## 2. MVP 정의 (MVP Definition)

### 2.1 한 문장 정의

> **"담당자가 수량·단가와 날짜만 입력하면 최신 제비율·부가세·합계·적용법령이 자동 검증된 원가설계서(도구 A)가 만들어지고, 그 데이터가 그대로 입찰공고문 초안(도구 B)으로 이어지며 AI가 누락 법령까지 추천하는 — 복사-수정 없이 검증부터 공고 작성까지 한 흐름으로 처리하는 스탠드얼론 계약업무 자동화 도구"**

문장 구성 요소와 기능의 대응:

| 문장 요소 | 대응 기능 | 해소 대상 |
|---|---|---|
| "수량·단가만 입력" | A-M1 (표준 항목·고정 로직) | 셀참조 깨짐, 양식 비표준 |
| "최신 제비율·부가세·합계 자동 검증" | A-M2~A-M4 | 지식의 개인화, 검증의 형식화 |
| "적용법령 자동 검증 + AI 누락 추천" | A-M5, B-M3 | 법령 판단의 개인 지식 의존 |
| "날짜만 입력 → 공고문 초안" | B-M1, B-M2 | 날짜 산재, 복사-수정 관행 |
| "복사-수정 없이 한 흐름으로" | 도구 A→B 파이프라인 | 근본원인 2 (복사-수정 워크플로우) |
| "스탠드얼론" | Phase 1 배포 제약 | — |

### 2.2 용도별 변형안

- **경영진 보고용 (임팩트 중심)**: "담당자 교체와 복사-수정 관행에서 발생하는 공고문·원가계산 오류를 구조적으로 차단하고, 연간 약 11,100시간(§5)의 업무를 절감하는 계약업무 자동화 도구"
- **엘리베이터 피치**: "공고문 작성부터 원가계산까지, 한 번 입력하면 알아서 채우고 스스로 검증하는 계약업무 도구"

### 2.3 MVP 범위와 성공 기준

| 항목 | 내용 |
|---|---|
| MVP 범위 | 도구 A Must(A-M1~A-M7) + 도구 B Must(B-M1~B-M6) = 릴리스 1~3차 완료 시점 |
| 범위 밖 | 나라장터 자동 등록·제출, 사내 기간계 시스템 연동, 유지보수 외 일반 공사유형 확장 |
| 성공 기준 ① 오류 예방 | 파일럿 기간 중 셀참조·제비율·부가세·날짜 오류 발생 0건 (도구 사용 건 기준) |
| 성공 기준 ② 시간 절감 | §5 추정치 대비 실측 — 도구 A 건당 2시간, 도구 B 건당 15분 단축 달성 여부 |
| 성공 기준 ③ 기준정보 재현성 | 산출물마다 제비율·법령·템플릿 버전, 적용 기준일, 갱신자/승인자 기록이 남을 것 |
| 성공 기준 ④ AI 추천 품질 | AI 법령추천의 채택률·수정률·기각률·오추천률을 측정하고, 추천 근거·출처·모델/프롬프트 버전을 기록할 것 |
| 성공 기준 ⑤ HWP 산출물 | MVP에서 HWP 내보내기를 Must로 유지하되, 지정된 표준 공고문 템플릿 1종에 대해 열림·서식·값 일치 검증을 통과할 것 |
| 성공 기준 ⑥ 보안 | 실데이터 파일럿 전 API 키 노출 방지, 외부 전송 데이터 마스킹, 사용 로그 기록 체계를 갖출 것 |
| 유의사항 | 위 한 문장 정의는 릴리스 3차(도구 B 완성) 기준. 1~2차 시점의 대외 커뮤니케이션은 "원가설계서 표준검증 도구"로 한정할 것 |

## 3. 문제정의 (Problem Definition)

### 문제 1. 입찰공고문 작성 오류

| 항목 | 내용 |
|---|---|
| **누가** | 계약업무 담당자 (2년 주기로 교체되는 비전담 인력, 본업과 겸직) |
| **언제** | 신규 입찰공고문을 작성할 때, 특히 기존(전임자) 공고문을 복사하여 날짜·기한 등을 수정하는 시점 |
| **무엇 때문에** | ① 제출기한 등 날짜 정보가 문서 여러 곳에 산재되어 있어, 복사한 문서에서 한 곳을 고치면 다른 곳은 누락됨<br>② 과거 사례를 "판단 참고용"으로 보다가 참고와 복사가 분리되지 않아 그대로 베끼는 경우 발생<br>③ 표준양식이 존재하지만 매번 빈 양식부터 채우는 것이 "실수할까봐" 심리적으로 더 불안하게 느껴져 기존 파일 복사를 선택<br>④ 결재 단계에서 대조할 체크리스트가 없어 결재자가 형식적 확인에 그침 |
| **어떤 불편** | 계약방식·날짜·기한 오기입 → 조달청 공고 취소 또는 재공고 → 사업 일정 지연, 대내외 신뢰도 저하 |
| **본인 추정 예방 효과** | 체크리스트 도입 시 약 50% 예방 가능 |

### 문제 2. 원가계산(일위대가) 오류

| 항목 | 내용 |
|---|---|
| **누가** | 현업부서 담당자 (원가계산 작성) 및 상위 직급자 (검토·결재) |
| **언제** | 전임자의 기존 원가계산 엑셀시트를 복사하여 값을 바꿔 넣는 시점, 그리고 계약 전 내부검토 단계 |
| **무엇 때문에** | ① 일위대가 작성 양식이 공사마다 상이하여 표준화된 기준이 없음<br>② 전임자 시트를 복사하는 과정에서 행/열 삽입·삭제로 셀 참조(SUM 범위 등)가 깨짐 (수식오류의 가장 큰 원인, 빈도 순: 셀참조 깨짐 > 안 맞는 옛 수식 재사용 > 입력값 오류)<br>③ 제비율이 6개월마다 개정되는 "동적 정보"인데, 이를 개인이 매번 능동적으로 확인·갱신해야 하는 구조 → 시기 누락 발생<br>④ 검토 단계에서 물량이 방대하여 처음부터 재계산이 아닌 "최종 금액이 그럴듯한지"만 훑어보는 수준에 그침 |

| **조직적 임팩트** | **3개 문제 중 가장 심각 (예산낭비 직결)** |

### 문제 3. 견적서 → 원가계산서 입력 오류

| 항목 | 내용 |
|---|---|

| **언제** | 업체로부터 받은 견적서(이미지/PDF)를 보고 원가계산서에 금액을 옮겨 적는 시점 |

## 4. 근본원인 분석 (Root Cause Analysis)

3개 문제를 관통하는 공통 구조:

1. **지식의 개인화** — 법령·제비율·판단기준이 시스템이 아니라 개인의 머릿속 또는 개인 파일에 존재 → 담당자 교체(2년 주기) 또는 기준 개정(제비율 6개월 주기) 시 정보가 낡거나 유실됨
2. **복사-수정이 사실상의 표준 워크플로우** — 표준양식이 있어도 실무에서는 기존 파일 복사가 더 안전하다고 느껴짐 → 복사 과정에서 구조적으로 오류가 전파(날짜 산재, 셀참조 깨짐)
3. **검증의 형식화** — 결재·검토 단계는 존재하나, 시간 부족과 대조 도구 부재로 "재계산"이 아닌 "훑어보기"에 그침 → 실질적 안전망 기능 상실
4. **겸직 구조** — 계약업무가 전담이 아니라 본업과 병행하는 짬짬이 업무 → 상시적 시간압박 속에서 처리

> **결론**: 이는 담당자의 실수(human error)가 아니라, **시스템이 사람에게 완벽한 기억력과 매번 처음부터 재계산할 시간을 요구하는 구조적 문제**이다.

## 5. 정량적 근거 (Quantitative Justification)

MVP 투자 타당성을 뒷받침하기 위해, 자동화 전/후 소요시간을 추정한다. (수치는 담당자 체감 추정치 기준)

### 5.1 도구 A — 원가계산서·견적서 검증

| 구분 | 계산 | 결과 |
|---|---|---|
| 현재 소요 | 400명 × 1건/월 × 6시간 | **2,400시간/월** |
| 효율화 후 | 400명 × 1건/월 × 4시간 (2시간 단축) | **1,600시간/월** |
| **절감** | | **800시간/월 (△33.3%)** → 연간 **9,600시간** |

### 5.2 도구 B — 입찰공고문 작성

## 6. 해결 방향 (Solution Direction)

**핵심 설계 원칙: "반복적 판단이 필요한 대조·검증·최신화는 자동화하고, 전문적 최종판단은 사람이 확인하는 구조로 설계한다"**

| 견적서 파일 처리 | PDF.js(뷰어), 이미지는 `<img>` 직접 렌더링 | OCR(A-S4)은 Should 단계에서 별도 검토 |
| 엑셀 내보내기 | SheetJS(xlsx) | 클라이언트단에서 파일 생성 가능 |
| PDF 내보내기 | jsPDF 또는 html2pdf.js | |
| 한글(hwp) 문서 내보내기 | **MVP Must 유지 — 기술 리스크 관리 대상** (10.5, §18 참조) | 표준 HWP 템플릿 1종을 기준으로 값 치환 방식 우선 검증, 실패 시 HWP 변환용 중간 포맷(DOCX/HTML) 및 수동 보정 절차를 병행 |
| 조달청 API 연동 (법령 유사사례 비교) | 조달청 공개 API(나라장터 등)로 공사명 키워드 검색 → 유사 공사 매칭 → 해당 공사의 적용 법령·조항 조회 | CORS 정책상 브라우저 직접 호출이 막힐 경우 서버리스 프록시 경유 필요 (S2 기술검증과 동일 이슈) |
| AI 법령추천 (B-M3) | Claude API(`/v1/messages`) 호출 | 실데이터 적용 전에는 서버리스 프록시 또는 보안 프록시 경유 원칙. 임시 외부 파일럿은 마스킹 데이터·개인 키 격리·사용량 제한 조건에서만 허용 |
| 데이터 저장 | 브라우저 세션 메모리 + JSON 파일 다운로드/업로드 (백엔드 DB 없음) | Phase 1 원칙과 일치 |
| 데이터 연계(A→B) | 같은 세션 내 메모리 전달 또는 JSON 파일 경유 | 스탠드얼론 2개 파일 간 연동 방식 사전 결정 필요 |

### 10.5 그 외 고려사항 (Other Considerations / Risks)

| 구분 | 내용 | 대응 방향 |
|---|---|---|
| **보안 — API 키 노출** | 클라이언트(브라우저)에서 Claude API를 직접 호출하면 API 키가 노출될 위험 있음 | 파일럿 단계는 개인 키로 한정 사용, 사내 배포 시 반드시 서버/프록시 경유로 전환 (Phase 2 필수 과제로 명문화) |
| **데이터 민감성** | 견적서에 업체명·단가 등 민감 정보 포함 | 파일럿은 더미데이터·마스킹 데이터 우선 사용, 실 데이터 사용 시 반출·저장 정책 사전 확인 |
| **AI 추천 신뢰성** | 법령 추천이 틀릴 경우 담당자가 그대로 신뢰해 오히려 새로운 오류 유형을 만들 위험 | 추천 근거를 반드시 함께 제시하고, "추천"이라는 표현을 명확히 사용하여 최종판단 책임이 담당자에게 있음을 UI·문서상 명시 |
| **한글(hwp) 포맷 리스크** | 조달청 제출용 공고문이 관행상 한글(hwp) 파일인 경우, 브라우저 기반 HWP 생성·편집은 기술 난도가 높음. 다만 본 MVP에서는 B-M4를 Must로 유지함 | 표준 HWP 템플릿 1종에 대한 값 치환 방식, DOCX→HWP 변환 방식, HTML→HWP 변환 방식 중 파일럿 가능한 방식을 기술검증한다. HWP 결과물은 열림·서식·값 일치 검증을 통과해야 하며, 실패 시 원인과 수동 보정 절차를 기록한다 |
| **감사 추적성** | AI 추천의 채택/기각 이력이 남지 않으면 추후 감사 시 판단 근거 확인 불가 | B-M5로 Must 반영. 추천 결과, 출처, 모델/프롬프트 버전, 담당자 판단 이력을 산출물 메타데이터와 별도 로그에 기록한다 |
| **파일럿 규모 제약** | 외부 환경 파일럿은 실제 조달청 시스템과 연동되지 않음 | 성공기준(§2.3)은 "도구 자체의 정확성·시간절감"에 한정, 실제 조달청 제출 프로세스와의 통합은 별도 과제 |

## 11. 데이터 스키마 (A→B 표준 연계 구조)

도구 A와 도구 B는 동일 세션 또는 JSON 파일을 통해 연결하되, 필드명을 사전에 고정하여 개발 단계 재작업을 줄인다.

### 11.1 원가설계 확정 데이터 스키마

| 데이터 그룹 | 필드 | 설명 |
|---|---|---|
| 프로젝트 기본정보 | `project_id`, `project_name`, `project_type`, `department`, `owner`, `created_at` | 공사/구매 기본 식별 정보 |
| 원본 업로드 | `source_file_name`, `source_file_hash`, `upload_at`, `mapping_status`, `unmapped_fields` | 업로드된 원본 엑셀(A) 추적 및 자동매핑 성공/실패 현황 |
| 원가 항목 | `item_code`, `item_name`, `quantity`, `unit`, `unit_price`, `amount`, `vat_type` | 수량·단가·금액·부가세 포함/별도 정보 |
| 제비율 | `rate_code`, `rate_name`, `rate_value`, `rate_version`, `effective_date`, `approved_by` | 적용 제비율 및 버전 정보 |
| 합계 검증 | `subtotal`, `vat_amount`, `total_amount`, `difference`, `validation_status` | 항목합·부가세·총계 정합성 |
| 법령 검증(규칙기반) | `required_laws`, `matched_laws`, `missing_laws`, `law_version` | 규칙 기반 필수 법령 매칭 결과 |
| 법령 검증(조달청 비교) | `reference_project_names`, `similarity_score`, `reference_applied_laws`, `comparison_result` | 조달청 API로 검색한 유사 공사명 목록·유사도·해당 사례 적용법령·규칙기반 결과와의 비교 결과 |
| 감사 로그 | `created_by`, `updated_by`, `confirmed_by`, `confirmed_at`, `change_log` | 작성·수정·확정 이력 (`confirmed_at` 시점 상태 = "통과") |

| 데이터 그룹 | 필드 | 설명 |
|---|---|---|
| 공고 일정 | `notice_date`, `bid_start_date`, `bid_deadline`, `opening_date`, `contract_period` | 날짜 단일 입력 및 다중 반영 대상 |

## 12. 기준정보 관리정책

기준정보는 담당자 개인 파일이 아니라 시스템의 통제 대상 데이터로 관리한다.

| 기준정보 | 관리 대상 | 갱신 주기 | 승인자 | 사용 기준 |
|---|---|---:|---|---|
| 제비율 | 공사/구매 유형별 제비율, 적용 기준일 | 반기 또는 개정 시 | 계약부서/감사 검토자 | 원가설계 작성일 또는 공고일 기준 중 사내 기준 확정 필요 |
| 법령 체크리스트 | 금액대·공사유형별 필수 조항 | 법령 개정 시 | 법무/감사 검토자 | 공고일 기준 적용 원칙 |
| 표준 항목 리스트 | 구매/유지보수 공사 표준 원가 항목 | 양식 개정 시 | 계약부서 | 도구 A 유형 선택 시 자동 로드 |
| 공고문 템플릿 | 표준 공고문 문구, 필수 입력 변수 | 양식 개정 시 | 계약부서/결재권자 | 도구 B 산출물 생성 시 적용 |
| AI 프롬프트 | 법령 추천 프롬프트, 금지 문구, 출력 형식 | 품질 개선 시 | 프로젝트 오너/감사 검토자 | 프롬프트 버전 기록 필수 |

### 12.1 기준정보 변경 원칙

1. 모든 기준정보는 버전, 적용일자, 갱신자, 승인자를 기록한다.
2. 기준정보 변경 전후의 산출물 계산 결과가 달라질 경우 변경 이력을 남긴다.
3. 과거 산출물은 당시 기준정보 버전으로 재현 가능해야 한다.
4. 기준정보가 미승인 상태이면 최종 공고문 내보내기를 제한한다.

MVP가 스탠드얼론 방식이라도 업무 상태와 책임 구분은 화면과 로그에 반영한다.

| 역할 | 주요 권한 | 제한 사항 |
|---|---|---|

| 결재자 | 최종 확정, 예외승인, 산출물 승인 | AI 추천 미검토 상태에서는 확정 불가 |

## 18. HWP 내보내기 구현·검증 계획 (B-M4 Must)

B-M4는 MVP Must로 유지한다. 단, HWP는 일반 브라우저 환경에서 생성·편집 난도가 높으므로 “기능 구현”과 “업무 사용 가능 수준 검증”을 분리해 관리한다.

### 18.1 구현 후보 방식

| 방식 | 설명 | 장점 | 리스크 |
|---|---|---|---|
| HWP 템플릿 값 치환 | 표준 HWP 파일 내 지정 변수 영역을 치환 | 실제 업무 양식과 가장 유사 | HWP 구조 파싱·치환 난도 높음 |
| DOCX 생성 후 HWP 변환 | DOCX를 생성하고 한글 또는 변환 도구로 HWP 변환 | 구현 난도 상대적으로 낮음 | 변환 과정에서 서식 깨짐 가능 |
| HTML/RTF 중간 포맷 | 브라우저에서 문서 생성 후 한글에서 열 수 있는 포맷으로 제공 | 스탠드얼론과 친화적 | 엄밀한 HWP 산출로 보기 어려울 수 있음 |
| 수동 보정 연계 | 자동 생성된 PDF/DOCX와 변수 매핑표를 제공해 HWP에 반영 | 실패 시 업무 연속성 확보 | 완전 자동화 효과 제한 |

### 18.2 MVP 채택 기준

1. 표준 공고문 HWP 템플릿 1종을 기준으로 검증한다.
2. 공고명, 금액, 일정, 참가자격, 법령 조항 등 핵심 변수값이 원본 데이터와 100% 일치해야 한다.
3. 한글 프로그램에서 파일이 정상적으로 열려야 한다.
4. 표·문단·날짜·금액 표기가 업무 사용 가능 수준으로 유지되어야 한다.
5. HWP 생성 실패 또는 서식 깨짐 발생 시 오류 메시지와 수동 보정 절차를 제공한다.
6. HWP 자동생성 방식의 한계는 파일럿 결과 보고서에 명시한다.

## 21. 데이터 흐름 상세설계 (End-to-End Data Flow)

본 섹션은 "① 엑셀 업로드 → ② 원가계산서 검증(견적서·법령·제비율·수식) → ③ 통과 판정 → ④ 입찰공고문 자동생성"으로 이어지는 전체 흐름을 단계별로 상세화한다.

### 21.1 전체 흐름 개요

```
[사용자] 원가계산서 파일 A (.xlsx) 업로드
        │
        ▼
[A-M0] 표준 스키마 자동 매핑 ──(매핑 실패 항목)──▶ 수동 매핑 화면(A-S0)
        │ (매핑 완료)
        ▼
[A-M1~A-M4] 원가설계서 검증
   ├─ 견적서 대조 검증 (A-M3): 업로드/입력된 견적금액 vs 원가계산서 항목금액, 부가세 이상치 감지
   ├─ 제비율 적용 검증 (A-M2): 내장 제비율 최신값과 파일 내 적용값 비교
   ├─ 수식(계산 로직) 검증 (A-M4): 소계·총계 재계산 후 파일 내 기재값과 정합성 비교
   └─ 적용법령 검증 (A-M5)
        ├─ 규칙 기반: 금액대·공사유형별 필수조항 매칭
        └─ 조달청 API 연동: 공사명 키워드로 유사 과거 공사 검색 → 해당 사례의 적용법령 조회 → 규칙기반 결과와 비교
        │
        ▼
[판정] 전 항목 통과? ──아니오──▶ 보완요청 (미통과 항목 하이라이트, 재입력 루프)
        │ 예
        ▼
[A-M6, A-M7] 계산근거 로그 기록 + 기준정보 버전 스탬프 → 상태 = "통과"
        │
        ▼
[A→B 전달] 확정 JSON (파일 A 통과본, §11.1 스키마) → 도구 B로 전달 (같은 세션 메모리 또는 JSON 파일 경유)
        │
        ▼
[B-M1] 통과본 데이터를 불러와 공고문 자동 초안 생성
        │
        ▼
[B-M2] 날짜/기한 입력 → 문서 내 다중 위치 자동 반영 + 논리 검증
        │
        ▼
[B-M3] AI(Claude API) 법령 누락·보완 추천 — 원가설계 데이터 + 조달청 유사사례 비교결과(21.1 상단)를 함께 프롬프트에 반영
        │
        ▼
[B-M5, B-M6] 추천 이력 기록 + 내보내기 차단조건 점검
        │
        ▼
[B-M4] 최종 공고문 산출 (엑셀/PDF/HWP)
```

### 21.2 단계별 상세

| 단계 | 입력 | 처리 | 출력 | 실패 시 처리 |
|---|---|---|---|---|
| ① 업로드 | 원가계산서 엑셀(A) | 파일 해시·업로드 시각 기록, 시트 구조 스캔 | `source_file_*` 메타데이터 | 지원되지 않는 형식(.xls 구버전 등)은 변환 안내 |
| ② 자동 매핑 | 엑셀 셀 데이터 | 헤더/열 패턴 매칭으로 표준 항목(§11.1)에 매핑 | 매핑완료 데이터 + `unmapped_fields` | 매핑 실패 항목은 A-S0에서 담당자가 열↔항목 수동 지정, 지정 이력 기록 |
| ③ 견적서 검증 | 매핑된 원가항목, 첨부 견적서(이미지/PDF) | 항목별 견적금액 대조, 부가세 포함/별도 이상치 탐지 | `validation_status`(항목별) | 불일치 항목 하이라이트, 부가세 미선택 시 다음 단계 진행 차단 |
| ④ 제비율 검증 | 매핑된 항목의 적용 제비율(파일 내 기재값) | 내장 최신 제비율(§12 기준정보)과 비교 | 일치/불일치 결과 | 불일치 시 최신값 자동 제안, 담당자 확인 후 반영 |
| ⑤ 수식 검증 | 매핑된 항목 수량·단가, 파일 내 소계·총계 | 항목합을 표준 로직으로 재계산 후 파일 기재값과 대조 | `subtotal`, `total_amount`, `difference` | `difference ≠ 0`이면 통과 불가, 항목 단위로 원인 표시 |
| ⑥ 법령 검증(규칙) | 계약금액, 공사/구매 유형 | 금액대·유형별 필수조항 매칭 | `matched_laws`, `missing_laws` | 누락 조항 표시, 보완 요청 |
| ⑦ 법령 검증(조달청 비교) | 공사명(프로젝트명 텍스트) | 조달청 API에 공사명 키워드로 검색 → 유사도 상위 N건 조회 → 각 사례의 적용법령 수집 | `reference_project_names`, `similarity_score`, `reference_applied_laws`, `comparison_result` | API 무응답·결과없음 시 규칙기반 결과만으로 진행하되 "유사사례 비교 미수행" 플래그 기록 |
| ⑧ 통과 판정 | ③~⑦ 결과 종합 | 전 항목 통과 여부 판정 | `validation_status = 통과`, 감사로그(A-M6) | 미통과 항목이 있으면 확정 버튼 비활성화 |
| ⑨ A→B 전달 | 확정된 §11.1 스키마 데이터 | JSON 직렬화, 같은 세션 메모리 또는 파일 전달 | 도구 B 입력값 | 전달 실패 시 수동 JSON 다운로드/업로드로 대체 |
| ⑩ 공고문 생성 | 통과본 데이터 | 표준 공고문 템플릿에 값 자동 채움 | 공고문 초안 | 필수값 누락 시 해당 위치 미채움 표시(자동 진행하지 않음) |
| ⑪ AI 법령추천 | 통과본 데이터 + 조달청 유사사례 비교결과 | Claude API 호출, 추천 법령·근거·출처 생성 | `recommended_laws`, `recommendation_reason`, `source_reference` | API 실패 시 규칙기반 결과만 표시, 재시도 옵션 제공 |
| ⑫ 최종 산출 | 승인된 공고문 데이터 | 내보내기 차단조건(§17) 점검 후 문서 생성 | 최종 공고문(엑셀/PDF/HWP) | 차단조건 미해소 시 산출 불가, 사유 표시 |

### 21.3 조달청 API 법령 유사사례 비교 — 상세 로직

1. 업로드된 원가계산서의 `project_name`(공사명)에서 핵심 키워드를 추출한다 (예: "청사 노후 배관 교체공사" → "배관 교체", "노후 시설").
2. 조달청 API에 키워드 검색을 요청하여 과거 공고·계약 사례 목록을 조회한다.
3. 사례명과의 텍스트 유사도(`similarity_score`)를 계산하여 상위 N건을 선정한다.
4. 선정된 사례들의 적용 법령·조항을 수집하여 `reference_applied_laws`에 기록한다.
5. 규칙 기반 결과(`matched_laws`/`missing_laws`)와 조달청 사례 기반 결과(`reference_applied_laws`)를 비교하여 `comparison_result`에 "일치" / "규칙기반에만 존재" / "사례에만 존재(검토 필요)"로 분류한다.
6. "사례에만 존재" 항목은 A-S4(검증결과 요약)에서 별도 강조 표시하고, 최종적으로 B-M3(AI 추천) 프롬프트의 입력 근거로도 함께 전달한다.

> **기술 제약**: 조달청 API의 실제 검색 정확도(공사명 표기 편차, 유사도 임계값)는 사전 검증이 필요하다. 브라우저에서 직접 호출 시 CORS 제한이 있을 수 있으므로, §10.2의 서버리스 프록시 검토 대상에 조달청 API 호출도 포함한다.

### 21.4 상태 값 갱신 (§15.1 상태 흐름과의 연계)

`업로드 → 매핑중 → 매핑완료 → 검증중 → (통과 | 보완요청) → 확정("통과") → 공고문생성중 → AI추천검토 → 최종확정 → 산출물 내보내기`

- 기존 §15.1의 `작성중 → 1차 검증완료 → …` 흐름 앞에 `업로드 → 매핑중 → 매핑완료` 상태가 추가된 것으로 이해하면 된다.
- "통과" 상태부터는 §11.1 스키마가 불변(immutable) 처리되며, 이후 수정 시 새 버전으로 분기하고 변경 이력을 남긴다 (§12.1 원칙 2, 3과 동일).
